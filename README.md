# Ellie — Discord channel mute bot

Модераторский бот для **временного запрета отправки сообщений** в конкретном текстовом канале (один сервер). Спецификация: [TODO_0_1.md](TODO_0_1.md).

## Требования

- Python 3.12+
- Discord bot token
- Права бота на сервере: **Управление каналами** (`Manage Channels`) и **Управление ролями** (`Manage Roles`, нужно для правки прав канала/веток), роль бота **выше** наказываемых пользователей

## Установка

```bash
python -m venv .venv
.venv\Scripts\activate   # Windows
pip install -r requirements.txt
```

Скопируйте примеры конфигурации:

```bash
copy config\config.yaml.example config\config.yaml
copy config\roles.yaml.example config\roles.yaml
copy .env.example .env
```

Заполните:

| Файл | Поля |
|------|------|
| `.env` | `DISCORD_TOKEN` |
| `config/config.yaml` | `guild_id`, `moderator_commands_channel_id`, `bot_logs_channel_id`, `database_path`, `log_level`, опционально `mention_gif_*` |
| `config/roles.yaml` | `role_ids` — ID ролей модераторов **сверху вниз** (старшие → младшие); старшая роль может наказывать младшую |

## Запуск

```bash
python bot.py
```

Slash-команды синхронизируются **только с guild** из `guild_id` (мгновенно после запуска).

## Автозапуск через systemd (Linux)

Для стабильной работы в фоне рекомендуется запускать бота как `systemd` service.
Это освобождает консоль и автоматически перезапускает процесс после падения
или рестарта сервера.

В репозитории есть шаблон юнита:
`deploy/systemd/ellie-bot.service.example`

### 1) Скопировать и отредактировать unit

```bash
sudo cp deploy/systemd/ellie-bot.service.example /etc/systemd/system/ellie-bot.service
sudo nano /etc/systemd/system/ellie-bot.service
```

Обязательно проверьте/замените:
- `User` / `Group`
- `WorkingDirectory`
- `ExecStart` (путь к Python в venv и `bot.py`)

### 2) Включить и запустить сервис

```bash
sudo systemctl daemon-reload
sudo systemctl enable ellie-bot.service
sudo systemctl start ellie-bot.service
```

### 3) Проверить состояние и логи

```bash
sudo systemctl status ellie-bot.service
journalctl -u ellie-bot.service -f
```

Полезные команды:

```bash
sudo systemctl restart ellie-bot.service
sudo systemctl stop ellie-bot.service
```

## Команды

| Команда | Где вызывать |
|---------|----------------|
| `/mute_user` | Текстовый канал/ветка наказания или бот-команды + `channel` |
| `/unmute_user` | Текстовый канал/ветка или бот-команды + `channel` |
| `/active_mutes` | Только бот-команды (`user`) |
| `/mute_help` | Только бот-команды |

В публичных каналах ответы бота **ephemeral** (видит только модератор). Действия пишутся в канал **логов ботов**.

### Каналы и ветки (`scope`)

`/mute_user` и `/unmute_user` принимают параметр `scope`:

| `scope` | Действие |
|---------|----------|
| `чат` | Запрет только в самом текстовом канале |
| `ветки` | Запрет во **всех ветках** канала; сам канал не затрагивается |
| `чат и ветки` | Запрет и в канале, и во всех его ветках |

- Если `scope` не указан: вызов в канале → `чат`, вызов в ветке → `ветки`.
- Наказание, выданное в ветке, изменяет права **родительского текстового канала** и распространяется на **все его ветки** (форум-ветки не поддерживаются).
- Запреты «чат» и «ветки» **независимы**: имеют отдельные сроки и снимаются раздельно по `scope`.
- Под капотом: `scope=чат` снимает `send_messages`, `scope=ветки` — `send_messages_in_threads`, `scope=чат и ветки` — оба бита.

## Устойчивость (БД и Discord)

| Ситуация | Поведение |
|----------|-----------|
| Пользователь **вышел** с сервера, срок ещё не истёк | Запись в БД и deny в канале сохраняются до `expire_at` |
| Пользователь **вышел**, срок истёк | Авто-снятие снимает overwrite по `user_id` (без Member), удаляет запись, пишет в лог-канал |
| **Аккаунт удалён** | Запись удаляется из БД; overwrite снимается best-effort; отдельного audit «снято» нет |
| Временный сбой Discord API | Повторы (3 попытки × 3 раунда с backoff 30s / 120s / 300s); **запись в БД не удаляется** |
| Постоянная ошибка (например нет прав) после всех попыток | Запись **остаётся** в БД; deny в Discord не трогается — нужно вмешательство модератора |
| **Перезапуск** процесса бота | Один раз при первом `on_ready`: просроченные mutes обрабатываются, активные перепланируются |
| **Reconnect** Discord без рестарта | Повторный `restore_all` **не** вызывается; таймеры в памяти продолжают работу |

Рассинхрон: overwrite без записи в БД бот **не снимает** через `/unmute_user`, пока запись не создана. **`/mute_user`** на уже замьюченного вручную пользователя создаёт запись в БД (усыновление) и дальше управляет сроком и снятием (в т.ч. ветки). Запись в БД без deny — при unmute/авто-снятии БД очищается.

SQLite: режим **WAL**, `busy_timeout` 5 с (см. `database/database.py`).

## Ответ GIF на @упоминание

При `@упоминании` бота в текстовом канале или ветке (кроме каналов логов и бот-команд) бот отвечает **reply** со случайным `.gif` из `assets/mention_gifs/`.

| Параметр в `config.yaml` | По умолчанию | Описание |
|--------------------------|--------------|----------|
| `mention_gif_enabled` | `true` | Включить/выключить фичу |
| `mention_gif_cooldown_seconds` | `600` | Минимальный интервал между ответами на **весь сервер** (хранится в БД) |
| `mention_gifs_dir` | `assets/mention_gifs` | Папка с файлами `.gif` |

Нужны права бота в канале: просмотр, отправка сообщений, прикрепление файлов. Без прав или на cooldown бот **молчит** (без ответа и без текста).

## Структура проекта

```
bot.py                 # Точка входа
core/                  # Конфиг, права, ответы, контекст каналов
database/              # SQLite, модели, миграции
modules/channel_mutes/ # Mute, scheduler, user_presence, unmute_outcome, audit
modules/mention_gif/   # Ответ GIF на @упоминание бота
assets/mention_gifs/   # Пул .gif для ответа (добавьте файлы на сервере)
config/                # config.yaml, roles.yaml
logs/                  # Файловые логи (создаётся автоматически)
```

### Добавление нового модуля

1. Создайте пакет в `modules/<name>/` (commands, service, repository).
2. Зарегистрируйте cog в `bot.py` → `setup_hook`.
3. Добавьте миграции в `database/migrations.py` при необходимости.
4. Опишите команды в README.

- [ ] Повторный mute — «продлил» в логах, новый срок
