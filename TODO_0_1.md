Ты опытный Python-разработчик. Создай полностью рабочий Discord-бот проект с чистой модульной архитектурой, рассчитанной на дальнейшее расширение функционала.

Технологии:

* Python 3.12+
* discord.py (slash-команды)
* SQLite
* asyncio
* logging

Язык интерфейса: все сообщения бота, slash-описания, DM и ответы об ошибках — **на русском**.

Основная задача первой версии:
Модераторский бот для временного запрета отправки сообщений пользователю в конкретном текстовом канале Discord.

## Область развёртывания

* Бот рассчитан на **один Discord-сервер (guild)**.
* Конфигурация **глобальная для инстанса** (один `config.yaml`, один `roles.yaml`): отдельные настройки per-guild **не требуются** в v1.
* В `config.yaml` указать **`guild_id`** целевого сервера; при старте проверять наличие guild. Если interaction с другого сервера или конфиг неверен — ответ модератору: **неверный `guild_id` в конфигурации бота** (не выполнять команду).
* Секреты: **`DISCORD_TOKEN` в `.env`**; в `config.yaml` токена нет.
* Slash-команды регистрировать **только на `guild_id`** из конфига (guild sync).
* **`moderator_commands_channel_id`** — канал «бот-команды» (только модераторы): `/active_mutes`, `/mute_help`; `/mute_user` и `/unmute_user` с **обязательным** `channel`.
* **`bot_logs_channel_id`** — канал логов ботов: публичная лента действий модерации для всей команды модерации.
* Поле `guild_id` в БД сохранять для целостности данных и возможного расширения; фактически все записи относятся к одному серверу из конфига.

## Механизм mute в Discord

* Только **обычные текстовые** каналы (`discord.TextChannel`). Голосовые, stage, forum, announcement — **вне scope** v1; при вызове не из текстового канала — отказ «только текстовые каналы».
* v1: только бит **`send_messages: deny`**. Архитектура: список deny-битов в одном месте (`PERMISSIONS_TO_DENY`), чтобы позже та же команда могла забирать несколько прав сразу.
* **Merge:** в существующий overwrite пользователя в канале добавляется только `send_messages: deny`, остальные биты не трогаем.
* **`overwrite_snapshot` (JSON):** только биты, которые меняет бот (v1: состояние `send_messages` **до** нашего deny — allow/deny/null). При первом mute без overwrite — `null` или явная метка «overwrite не было».
* **Снятие:** убрать **только** наш `send_messages` deny; остальные биты overwrite не трогать. Восстановить значение `send_messages` из snapshot (только то, что менял бот). Если overwrite создан нами целиком и после снятия битов не осталось — удалить overwrite.
* Роль бота **выше** наказываемого; иначе — **только ephemeral модератору**, без записи в `bot_logs_channel_id`.

Подробности — в «Принятые решения».

## Права на команды модерации

Команды модерации доступны, если выполняется **хотя бы одно** условие:

* у участника есть право Discord **`Administrator`** на сервере (в т.ч. роли «модератор» с ограниченным, но полным флагом Administrator);
* **или** у участника есть одна из ролей из `roles.yaml` (список `role_id` — штатные модераторы без полного Administrator).

Проверка прав — в `core/permissions.py`:

* `can_moderate(member)` — доступ к slash-командам;
* `can_mute_target(moderator, target)` — можно ли наказать цель (см. «Принятые решения»).

## Каналы сервера (конфиг)

| Ключ в `config.yaml` | Назначение |
|----------------------|------------|
| `moderator_commands_channel_id` | «Бот-команды» — `/active_mutes`, `/mute_help`; mute/unmute с явным `channel` |
| `bot_logs_channel_id` | Логи ботов — записи о выдаче/снятии наказаний (и авто-снятии по таймеру) |

**Немодераторский канал** — любой текстовый канал, кроме `moderator_commands_channel_id` и `bot_logs_channel_id`.

**`bot_logs_channel_id`** — только сообщения от ботов (аудит). Любые slash-команды модуля в этом канале **запрещены**.

## Журнал в канале логов ботов

При **ручной** выдаче или снятии наказания (`/mute_user`, `/unmute_user`) бот отправляет сообщение в `bot_logs_channel_id` (не ephemeral).

Формат (русский; участники — **упоминания** `<@user_id>` с отображаемым `display_name` на сервере):

* выдача: `{модератор}` **запретил** пользователю `{пользователь}` общаться в чате `{#канал}`
* продление: `{модератор}` **продлил** запрет пользователю `{пользователь}` в чате `{#канал}`
* снятие: `{модератор}` **снял запрет** пользователю `{пользователь}` общаться в чате `{#канал}`

Дополнительно: срок, причина (если есть), **`user_id`**, **`channel_id`**, **`moderator_id`** (для разбора споров).

При **автоматическом** снятии по таймеру (всегда пытаться отправить, даже если канал/участник удалены — иначе только файловый лог):

* `{имя_бота_на_сервере}` **снял запрет** пользователю `{пользователь}` общаться в чате `{#канал}` (истёк срок) + ID

Ошибка отправки в лог-канал **не отменяет** наказание/снятие; записать в файловый лог `logs/`.

Реализация: `modules/channel_mutes/audit_log.py` или метод в `service.py`, вызываемый из единой точки после успешного mute/unmute.

## Ответы модератору (успех и ошибки)

На **любой** вызов модераторской slash-команды бот **обязан** ответить **от своего имени** вызвавшему модератору — и при успехе, и при ошибке. «Тихий» отказ без сообщения **запрещён**.

Канал ответа — тот же, где вызвана команда (через interaction), не DM модератору.

### Видимость ответа

| Где вызвана команда | Ответ бота модератору |
|---------------------|------------------------|
| Немодераторский канал | **ephemeral** (видит только модератор) |
| `moderator_commands_channel_id` | обычное сообщение (видят модераторы в «бот-командах») |

Запись в `bot_logs_channel_id` — только при **успешном** mute/unmute, не при ошибках валидации.

### Обязательные ответы при ошибках (текст на русском, понятный модератору)

Единый хелпер `reply_moderator(interaction, text, *, success=False)` в `core/responses.py` (или аналог) — **все** ветки, включая `try/except` на уровне команды.

| Ситуация | Пример смысла сообщения |
|----------|-------------------------|
| Неверный формат длительности (`10x`, пусто, 0, > 28 дней) | «Неверный срок. Используйте формат: 10m, 2h, 3d (максимум 28 дней).» |
| Пользователь не найден (нет на сервере / неверный ID) | «Пользователь не найден на этом сервере.» |
| Не указан обязательный `channel` в «бот-командах» | «Укажите канал: параметр channel обязателен в этом чате.» |
| Команда в `bot_logs_channel_id` или не тот чат для `/active_mutes`, `/mute_help` | «Эту команду нельзя использовать здесь. …» |
| Неверный `guild_id` в конфиге / interaction не с того сервера | «Неверный guild_id в конфигурации бота.» |
| Вызов не из текстового канала (где применимо) | «Команда доступна только в текстовых каналах.» |
| У вызывающего нет прав модерации | «У вас нет прав для этой команды.» |
| Цель наказания запрещена политикой | «Нельзя применить наказание к этому участнику.» |
| У бота нет прав / роль ниже цели | «Не могу выдать наказание: …» — **только модератору**, без `bot_logs` |
| Целевой канал — «бот-команды» или лог-канал | «Нельзя выдать наказание в этот канал.» |
| При `/unmute_user`: пользователь **не наказан** в выбранном канале | «Пользователь не ограничен в общении в канале #название.» |
| Внутренняя ошибка, Discord API, необработанное исключение | «Не удалось выполнить команду. Попробуйте позже или обратитесь к администратору.» + полный traceback в файловый `logs/` |
| Таймаут / сбой при обращении к Discord | то же, без падения процесса бота |

При успехе — краткое подтверждение, например: «Наказание выдано: @user, канал #…, до …» / «Запрет снят: …».

Ошибки валидации **до** изменений в Discord и БД. При частичном сбое после изменений — сообщение модератору + запись в `logs/`; по возможности откат или идемпотентное завершение (см. «Надёжность»).

## Приватность ответов

См. таблицу в «Ответы модератору»: в публичных каналах все ответы (успех и ошибки) — **ephemeral**; в «бот-командах» — обычные.

Ограничение Discord: факт вызова slash другим участникам API не всегда скрывает; цель v1 — **не показывать текст ответа бота** остальным в публичных каналах.

DM наказанному — отдельно, только при успешном mute/unmute.

## Требования к функционалу

### 1. Команда выдачи наказания: `/mute_user`

Имена параметров slash — **английские**; описания параметров и тексты ответов — **русские**.

Параметры:

* `user` (обязательно) — выбор участника из списка **или** строка с numeric `user_id`
* `duration` (обязательно; `10m`, `2h`, `3d` — только суффиксы **m, h, d**)
* `reason` (необязательно)
* `channel` — целевой текстовый канал (см. правила ниже)

Ограничения:

* минимум **1 минута** (60 секунд);
* максимум **28×24 часа** от момента выдачи/продления (`expire_at = now_utc + duration`);
* целевой канал — обычный текстовый, не `moderator_commands_channel_id` и не `bot_logs_channel_id`;
* проверка `can_mute_target` до изменений в Discord.

**Где вызывать и параметр `channel`:**

| Место вызова | Параметр `channel` | Канал наказания |
|--------------|-------------------|-----------------|
| Немодераторский текстовый канал | **необязателен** (по умолчанию — канал вызова) | Указанный `channel` или текущий канал |
| `moderator_commands_channel_id` («бот-команды») | **обязателен** | Только из параметра `channel` |

Поведение:

* ограничение применяется в выбранном канале наказания (не в канале вызова, если команда из «бот-команд»);
* порядок: **Discord (overwrite) → БД → планировщик**; при сбое после Discord — **откат** overwrite по snapshot;
* DM по краткому шаблону (см. «Принятые решения»); при продлении — **новое** DM; если DM закрыты — не прерывать, залогировать;
* запись в БД + планировщик;
* запись в `bot_logs_channel_id` (см. «Журнал в канале логов ботов»);
* файловое логирование в `logs/`;
* любой исход — ответ модератору от имени бота (см. «Ответы модератору»).

**Повторное наказание** (тот же `guild_id` + `channel_id` + `user_id`):

* **не создавать** вторую строку в БД;
* **`expire_at` = сейчас (UTC) + новая длительность** (не от старого срока);
* **не менять** `overwrite_snapshot`;
* **обновить** `moderator_id`, `reason`, `created_at` (время последнего продления);
* пересоздать задачу планировщика;
* в `bot_logs_channel_id` — текст **«продлил»** (не «запретил»);
* новое DM пользователю.

### 2. Команда снятия наказания: `/unmute_user`

Параметры: `user`, `channel` (правила как у mute).

Ограничения:

* **все** slash-команды модуля **запрещены** в `bot_logs_channel_id`;
* целевой канал не должен быть `moderator_commands_channel_id` или `bot_logs_channel_id`.

**Где вызывать и параметр `channel`** (аналогично `/mute_user`):

| Место вызова | Параметр `channel` | Канал снятия |
|--------------|-------------------|--------------|
| Немодераторский текстовый канал | **необязателен** (по умолчанию — канал вызова) | Указанный `channel` или текущий канал |
| `moderator_commands_channel_id` («бот-команды») | **обязателен** | Только из параметра `channel` |

Поведение:

* проверка: запись в БД **и** состояние overwrite в Discord; при расхождении — привести к результату команды (см. «Принятые решения»);
* если в БД **нет** записи — «не наказан», БД/overwrite не трогать (кроме случая «в БД нет, чужой deny» — **игнорируем** чужой deny);
* если в БД есть, в Discord overwrite **нет** — считать уже снятым, **очистить БД**, отменить планировщик;
* если наказание активно — Discord (снять наш deny) → БД → планировщик; при сбое после Discord — откат по snapshot;
* DM пользователю о досрочном снятии (ошибка DM не ломает команду);
* запись в `bot_logs_channel_id`;
* операция **идемпотентна** при повторном снятии (см. «Надёжность»);
* любой исход — ответ модератору от имени бота (см. «Ответы модератору»).

### 3. Команда просмотра наказаний пользователя: `/active_mutes`

* Только в `moderator_commands_channel_id`.
* Параметр **`user`** (обязательно) — участник или `user_id` строкой.
* Ответ: список каналов, где у пользователя **активно** наказание в БД: `#канал`, `expire_at`, `reason`; если записей нет — «У пользователя нет активных ограничений в каналах».
* Не из «бот-команд» — ephemeral с подсказкой; в «бот-командах» — обычное сообщение.

### 4. Справка: `/mute_help`

Смысл: «mute channel help» — справка по командам ограничений в каналах.

* Имя slash-команды в Discord: **`/mute_help`** (отдельная команда; `/mute_user` уже занята выдачей наказания).
* Вызывается **только** в `moderator_commands_channel_id` («бот-команды»).
* Параметров нет.
* Доступ: те же права, что у остальных модераторских команд (`can_moderate`).
* Ответ: **обычное текстовое сообщение** от бота в канале (не ephemeral), чтобы справка оставалась в чате для всех модераторов.
* При вызове не из «бот-команд» — **ephemeral**: «Справка доступна только в канале #бот-команды».

**Содержимое сообщения** (русский, структурированный текст или embed с `description`; без пинга @everyone):

1. Краткое описание модуля (запрет писать в конкретном канале, остальные каналы доступны).
2. Таблица/список команд с **где вызывать** и **примером**:

| Команда | Где использовать | Пример |
|---------|------------------|--------|
| `/mute_user` | В канале наказания **или** «бот-команды» + `channel` | `/mute_user user: @User duration: 2h reason: спам` |
| `/mute_user` | «Бот-команды», `channel` обязателен | `/mute_user user: @User duration: 1d channel: #общий` |
| `/unmute_user` | В канале снятия или «бот-команды» + `channel` | `/unmute_user user: @User` |
| `/active_mutes` | Только «бот-команды», нужен `user` | `/active_mutes user: 123456789012345678` |
| `/mute_help` | Только «бот-команды» | `/mute_help` |

3. Примечания: макс. срок 28d; форматы `10m`, `2h`, `3d`; логи действий — канал `bot_logs_channel_id`; ответы в публичных каналах — только модератору (ephemeral).

Текст справки вынести в константу или `config/help_text.md` / `channel_mutes/help_text.py` для правки без смены логики команд.

## Хранение данных (SQLite)

Таблица наказаний:

* `id` (PK)
* `guild_id`
* `channel_id`
* `user_id`
* `moderator_id`
* `reason` (nullable)
* `created_at` (UTC)
* `expire_at` (UTC)
* `overwrite_snapshot` (nullable, JSON) — только биты, которые меняет бот (v1: `send_messages` до deny)

Ограничения:

* **UNIQUE** (`guild_id`, `channel_id`, `user_id`) — одно активное наказание на пользователя в канале.

Все временные метки в БД — **UTC**, формат **ISO 8601** (единый по проекту).

## Планировщик наказаний

Не использовать cron. Внутренний планировщик на **asyncio**.

Требования:

* при выдаче/продлении — задача автоматического снятия по `expire_at`;
* при `/unmute_user` — отмена задачи;
* при старте бота — загрузить активные записи из БД и перепланировать;
* если `expire_at` уже в прошлом — снять наказание сразу при старте;
* корректная работа после перезапуска процесса;
* **одна asyncio-задача на запись** в БД (объём — сотни/низкие тысячи, не десятки тысяч одновременно).

Снятие по таймеру вызывает тот же метод сервиса, что и `/unmute_user` (единая точка логики), с записью в `bot_logs_channel_id` (авто-снятие, без модератора в тексте — см. «Журнал»).

## Надёжность и идемпотентность

* `service.unmute(...)` идемпотентен при повторном вызове.
* **Mute:** Discord → БД → планировщик; сбой после Discord — откат overwrite.
* **Unmute (ручной/авто):** Discord → БД → отмена задачи; сбой после Discord — откат по snapshot.
* Планировщик при 403/404: повторы, затем очистка БД — константа **`SCHEDULER_MAX_RETRIES = 3`** в модуле.
* Overwrite есть, записи в БД **нет** (чужой/ручной) — **не трогать** overwrite.

## Конфигурация

### `config.yaml`

* **`guild_id`** — ID единственного сервера;
* токен — только в **`.env`** (`DISCORD_TOKEN`);
* префикс (на будущее);
* путь к БД;
* уровень логирования;
* **`moderator_commands_channel_id`** — канал «бот-команды» (`/active_mutes`, `/mute_help`; `/mute_user` и `/unmute_user` — с обязательным `channel`);
* **`bot_logs_channel_id`** — канал логов ботов (лента действий модерации).

### `roles.yaml`

* список `role_id` модераторских ролей (в т.ч. роли с частичными правами администратора, если у них **нет** флага `Administrator` в Discord);

### Кого можно наказать (`can_mute_target`)

| Цель | Разрешено |
|------|-----------|
| Себя | **Нет** |
| Любой бот (включая этого) | **Нет** |
| Участник с `Administrator` | **Нет** |
| Владелец сервера | **Нет** |
| Участник **без** ролей из `roles.yaml` | **Да** (для любого модератора из списка) |
| Участник с ролью **ниже** в `roles.yaml` (ниже в файле = младше) | **Да** |
| Участник с той же или **более старшей** ролью из `roles.yaml` | **Нет** |
| Исключение | У модератора есть Discord **`Administrator`** — может наказать любого, кроме строк выше |

`roles.yaml`: порядок `role_ids` **сверху вниз** — от старшей модераторской роли к младшей.

`can_moderate`: `Administrator` **или** любая роль из `roles.yaml` — **да** (п. 4.2).

Ошибка иерархии бота: **только ephemeral модератору**, без `bot_logs`.
